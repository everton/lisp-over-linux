*STATUS: WIRED PATH WORKING end-to-end in QEMU.* Kernel half DONE (tag #21:
=CONFIG_NET= + =INET= + =virtio-net=). Userland half DONE: =sb-bsd-sockets= is
baked into the image; =eth0= is configured at boot by *DHCP* (=dhcp.lisp=, a real
DISCOVER/REQUEST handshake — static fallback if no server answers); and the
network REPL (=net.lisp=) serves the *full line editor* over TCP on =:4005= —
*ON by default* (toggle off from the supervisor menu with =t=), with a host-side
raw-forwarding client (=host-client/lol-repl-client.c=) so
arrows/history/Ctrl-C/Ctrl-D work remotely with no guest PTY (§6a). Verified
against QEMU SLIRP: eth0 leases =10.0.2.15= (gw =10.0.2.2=, dns =10.0.2.3=), and
the host client gives a live editing REPL — =(+ 40 2)= → =42=. *Still ahead:* an
off-subnet default route + DNS (§2c), and Wi-Fi for real hardware (§7). It is the
single biggest subsystem we have taken
on, because — unlike the framebuffer alien (pure userland) or the bigger console
font (pure kernel) — networking straddles *BOTH* the kernel and userland at once.

* The one-paragraph essence

To "have networking" two independent things must both become true: (1) the
KERNEL must be able to move packets — a socket layer (=NET=), a protocol stack
(=INET= = TCP/IP), a network device layer, and a *driver for an actual NIC*; and
(2) USERLAND must be able to *use* sockets — SBCL needs the =sb-bsd-sockets=
contrib baked into its frozen image, the interface needs an *IP address*
(static, or DHCP), and something has to resolve names (DNS) if we want them.
Neither half is useful without the other. We will build the kernel half first
(prove a link exists with packets), then the userland half (reach it from Lisp).

* Why this is harder than everything so far

| Past change            | Kernel? | Userland? | Notes                                   |
|------------------------+---------+-----------+-----------------------------------------|
| Draw the alien         | no      | yes       | /dev/fb0 already existed                 |
| Bigger fbcon font      | yes     | no        | compile-in font + cmdline               |
| *Networking*           | *yes*   | *yes*     | NET+INET+NIC driver AND sb-bsd-sockets  |

Everything we relied on before boots *early* and needs no driver negotiation.
A network link is different: a real driver must bind to a real (PCI) device,
bring the link up, and exchange packets with something on the other end. So we
test it in stages and prove each layer before adding the next.

* 1. Kernel half — what has to be turned on

*DONE in tag #21* — all of the below are now =y= (=grep -c '^CONFIG_NET=y'=
returns 1). The keystone =CONFIG_PCI= was already on (tag #19 for USB/ACPI), and
=CONFIG_PCI_MSI= too — modern NICs want MSI/MSI-X — so the bus plumbing was
already present. The options we turned on, smallest-useful-set first
(=CONFIG_IPV6= deliberately left OFF, see §5):

| Option                     | Role                                                        |
|----------------------------+-------------------------------------------------------------|
| =CONFIG_NET=               | The socket / protocol core — THE master switch              |
| =CONFIG_INET=              | IPv4 (TCP, UDP, ICMP). The thing we actually want           |
| =CONFIG_UNIX=              | AF_UNIX local sockets (handy even before a NIC works)       |
| =CONFIG_PACKET=            | AF_PACKET (raw sockets; some DHCP clients use it)           |
| =CONFIG_NETDEVICES=        | The network-device layer (gate for any NIC driver)          |
| =CONFIG_NET_CORE=          | Core netdev machinery                                        |
| a NIC driver               | the actual hardware — see stage plan below                  |
| (=CONFIG_IPV6=)            | optional; skip at first to stay minimal                     |

NIC driver — the staged target list (easiest → hardest):

1. *=CONFIG_VIRTIO_NET=* (+ =CONFIG_VIRTIO=, =CONFIG_VIRTIO_PCI=) — a QEMU
   paravirtual NIC. This is the *first* target: trivial to wire up in
   =build.sh --run= (=-netdev user,... -device virtio-net-pci,...=), reliable,
   and isolates "does our NET/INET stack work" from any real-hardware quirk.
2. *=CONFIG_E1000= / =CONFIG_E1000E=* — Intel emulated/real NIC; another easy
   QEMU device and present on many laptops/desktops.
3. *The laptop's actual NIC.* This machine (checked with =lspci -nnk=) has *no
   wired port at all* — only Intel Wi-Fi (=8086:34f0=, see §7). So on real
   hardware Wi-Fi is the *only* path to a network; wired/virtio is a QEMU-only
   convenience. Wi-Fi is a big jump and a separate, later project — its full
   shape (and how to add it without polluting this MIT repo or hard-coding one
   laptop) is §7.

Reminder (the frozen-heap / minimal rule): build NIC drivers *=y= (built-in)*,
not =m= — we ship no modules and have no =modprobe=.

* 2. Userland half — reaching the network from SBCL

A working kernel link is invisible until userland does three things:

** (a) sockets in the Lisp image — =sb-bsd-sockets= — *DONE*
SBCL's BSD-sockets API lives in the *=sb-bsd-sockets= contrib*, a =require=-loaded
fasl. Our image is a *frozen heap* (see [[file:sbcl-init.org][sbcl-init.org]]): we cannot =require= it at
runtime (no contrib fasls on the initramfs, no network to fetch them — the very
chicken-and-egg this whole file is about). So =build.sh= does
=(require :sb-bsd-sockets)= *at build time*, before =save-lisp-and-die=, with
=SBCL_HOME= pointed at the built tree's =obj/sbcl-home= so the contrib fasl
resolves. The socket code is then frozen into =/sbin/lisp= — at runtime no fasl
or =SBCL_HOME= is needed. Pure Lisp + kernel syscalls; no new native library.
(Resolved the old open question: yes, this SBCL tree can =require= it at build
time, given =SBCL_HOME=.)

** (b) an IP address — static, then DHCP — both *DONE*
A NIC with no address can't talk.
- *Static* (=bring-up-interface=, =net.lisp=): assigns address + mask and brings
  the link UP via =SIOCSIFADDR= / =SIOCSIFNETMASK= / =SIOCSIFFLAGS= ioctls on a
  throwaway UDP socket — the classic =ifconfig= path as =sb-alien= calls (no
  =ifconfig=/=ip= binary). Kept as the fallback if DHCP gets no answer.
- *DHCP* (=dhcp.lisp=): a real *DISCOVER → OFFER → REQUEST → ACK* conversation,
  ~a hundred lines of pure Lisp on a =sb-bsd-sockets= UDP socket. The BOOTP packet
  and option parsing are built/parsed by hand. Two tricks make it work with NO IP
  yet, over an ordinary socket (no =AF_PACKET=): *SO_BINDTODEVICE* ties the socket
  to =eth0= so the broadcast leaves the right NIC with no routing entry, and we set
  the DHCP *broadcast flag* so the server broadcasts its replies to
  =255.255.255.255:68= (receivable) instead of unicasting to an address we do not
  have. =configure-eth0= runs it at boot, applies the leased IP+mask, and stashes
  the gateway/DNS (=*gateway*= / =*dns*=) for §2c. Verified against QEMU SLIRP:
  eth0 gets =10.0.2.15=, gw =10.0.2.2=, dns =10.0.2.3= from the ACK.

** (c) name resolution + off-subnet routing — DNS (next)
DHCP already hands us the gateway and DNS server (=*gateway*= / =*dns*=). Applying
IP+mask gives an *on-link route* for the whole subnet, which is enough to reach
everything in QEMU SLIRP's =10.0.2.0/24= (gateway =10.0.2.2=, DNS =10.0.2.3=). Two
things remain for the *outbound* world: an *off-subnet default route* via the
leased gateway (=SIOCADDRT= / =struct rtentry=), and *DNS* itself — =getaddrinfo=
via glibc would drag in NSS plugin files, so cleaner to write a minimal
DNS-over-UDP query in Lisp (sending to =*dns*=), or just use raw IPs at first.

* 3. The staged bring-up plan (prove each layer before the next)

1. *Kernel builds with NET+INET+VIRTIO_NET* — *DONE (tag #21).* The NIC
   enumerates on PCI (=[1af4:1000]=, class 0x0200) and =show-net-interfaces=
   lists =eth0 (down)= at boot. =kernel-config.org= §7 + changelog updated; the
   QEMU NIC wiring is in =build.sh= (=QEMU_NET=).
2. *Static IP from Lisp* — *DONE.* =bring-up-interface= (=net.lisp=) brings =eth0=
   UP|RUNNING at =10.0.2.15/24=; =show-net-interfaces= now prints the address.
3. *First packets — a TCP server* — *DONE.* We went straight to the TCP REPL
   (§6a, =start-network-repl=): =bind= / =listen= / =accept= a connection and run
   the REPL over the socket stream. From the host, =nc 127.0.0.1 4005= (QEMU
   =hostfwd=) returns a live remote REPL — verified =(* 6 7)= → =42=. This proves
   the whole INET path (a UDP/ICMP "hello" or an outbound TCP *client* would be
   strictly easier and were skipped as redundant).
4. *DHCP* — *DONE.* =dhcp.lisp= does a real DISCOVER/REQUEST handshake at boot
   (=configure-eth0=), so =eth0= gets =10.0.2.15= dynamically instead of a hard
   coded address; gateway/DNS are parsed and stashed. Verified against QEMU SLIRP.
5. *(later)* off-subnet default route + DNS (§2c), then real-hardware Ethernet,
   then — much later — Wi-Fi (§7).

* 4. QEMU test wiring (for build.sh --run)

User-mode networking needs no root and no host bridge — ideal for our setup:

#+begin_src sh
  # add to the qemu line in build.sh --run:
  -netdev user,id=net0,hostfwd=tcp::5555-:23 \
  -device virtio-net-pci,netdev=net0
#+end_src

=user= (SLIRP) gives the guest =10.0.2.15=, gateway =10.0.2.2=, DNS =10.0.2.3=,
all in software — no privileges. =hostfwd= forwards a host port to a guest port
so we can later reach a Lisp TCP server from the host. This keeps the whole net
bring-up testable headless, exactly like the framebuffer screenshots.

* 5. What to NOT do (scope guards)

- *No Wi-Fi at first* — cfg80211/mac80211 + firmware blobs + WPA supplicant is a
  whole subsystem; wired/virtio first.
- *No modules* — built-in (=y=) drivers only; we have no module loader.
- *No glibc NSS / resolver files* — prefer tiny Lisp DNS over dragging NSS in.
- *No IPv6 initially* — IPv4 only keeps the surface small.
- Keep it *didactic*: a hand-rolled Lisp DHCP/DNS/echo teaches the protocols and
  fits the "Lisp machine drives the hardware via FFI" theme better than shipping
  opaque busybox applets (though busybox is the pragmatic fallback).

* 6. Exposing the REPL over the network

Once the userland half works (=sb-bsd-sockets= baked in, =eth0= has an IP), the
SBCL REPL can be reached remotely. Two routes, in increasing cost — *(a) is now
implemented* (=net.lisp=); (b) remains planning:

** (a) Raw TCP REPL — *IMPLEMENTED* (=net.lisp= + a host client)
A listening socket whose stream is bound to =*standard-input*= / =*standard-output*=
turns the REPL into a network REPL. How it actually ships:

- *ON at boot; a menu toggle.* =start-net-repl= / =stop-net-repl= /
  =toggle-net-repl= manage a background accept thread; =supervisor-main= starts it
  at boot, and the menu's =t= toggles it, showing =[RUNNING]= / =[stopped]=. (It is
  remote root — see the warning below — but acceptable here because the only wire
  that reaches it is the trusted QEMU =hostfwd=.) =stop-net-repl= must =shutdown=
  the listening socket and =join= the accept thread before returning, or a later
  re-enable hits =EADDRINUSE= (closing a socket does not wake a blocked
  =socket-accept= in another thread).
- *It serves the FULL line editor*, not =plain-repl=. The earlier worry was "a
  socket peer is not a terminal we can put in raw mode" — true, but it does not
  have to be: a tiny *host-side raw-forwarding client*
  (=host-client/lol-repl-client.c=, ~120 lines, any libc) puts the *host* terminal
  in raw mode and forwards keystrokes byte-by-byte, so =read-line-edited= receives
  escapes and drives history/editing remotely; the redraw escapes go back to the
  client. So we get full editing over a bare socket *with no PTY on the guest*
  (this is the design worked out in the discussion behind §6 — the client owns
  raw mode; the guest only needs a terminal if it runs *unmodified* programs,
  which our custom editor is not).
- *Ctrl-C / Ctrl-D* are forwarded raw and handled in =read-line-edited=: Ctrl-C
  aborts the current line (=:cancel=, fresh prompt — it never reaches the editor
  as a byte on the *local* console, where ISIG turns it into SIGINT), Ctrl-D on an
  empty line ends the session and the client disconnects cleanly.

Connect from the host: build the client once (=cc -O2 -o host-client/lol-repl-client
host-client/lol-repl-client.c=) then run =host-client/lol-repl-client= (defaults to
the =hostfwd= =127.0.0.1:4005=). Pure Lisp + syscalls on the guest; *no new kernel
option and no native binary in the image* — the client is host-only tooling (C, so
it needs no host runtime at all; deliberately a different language from the guest).

*The catch — this is plaintext and unauthenticated.* Anyone who can reach the
port gets a remote =eval= (i.e. remote root, since the REPL is PID 1). It is
only acceptable when *every wire that reaches the box is already trusted* — the
QEMU =hostfwd= case (bound to the host loopback) or a direct cable to a dev
machine. *Never bind it to an untrusted LAN.* Encryption cannot be added "for
free": it must run on whatever machine the untrusted wire touches — which leads
to (b).

** (b) Dropbear — an authenticated/encrypted alternative
If the box must sit on a network we do *not* control, the REPL needs real SSH in
front of it. *OpenSSH is too heavy* (drags in OpenSSL/libcrypto, PAM, NSS — all
against this project's spirit). The realistic choice is *Dropbear*: a tiny SSH
server built for embedded systems that *statically links its own bundled crypto*
into a single small binary (the "busybox of ssh"). What adopting it would cost:

- A new *static third-party binary* in the initramfs (a C program, not a library
  — consistent with our "no new native *libraries*" rule; links musl static like
  =preinit=).
- Kernel: *PTY support is already present* — =CONFIG_UNIX98_PTYS=y= is on (and
  devpts is folded into it on modern kernels), so =/dev/ptmx= exists. The only
  missing piece is *mounting devpts* (=mount -t devpts devpts /dev/pts=, plus a
  =/dev/pts= dir) in =preinit=, so a session can get a slave tty. No kernel
  rebuild needed for this. (The old *legacy* BSD PTYs were removed in tag #22 —
  =CONFIG_LEGACY_PTYS= off — to declutter =/dev=; Unix98 is what anything modern
  wants anyway.)
- A minimal *accounts / host-key* setup: a host key generated at first boot, and
  one user (SSH authenticates *something*).
- Point Dropbear's login program at =/sbin/lisp= instead of a shell — so logging
  in drops you straight into the SBCL REPL, encrypted and authenticated.

Sequencing: ship (a) first (it costs nothing beyond the socket work already in
the plan and is perfect for QEMU/dev), and only take on (b) when a real,
untrusted network is actually in play.

* 7. Wi-Fi — needed eventually, but out-of-tree and opt-in (planning)

We *will* need this. "Minimal" must not collide with "useful": a machine that
can't reach a network is almost a toy, and — see §1 — this laptop has no wired
port, so Wi-Fi is its only real link. So Wi-Fi is firmly on the roadmap; it is
deferred only because it cannot be tested in QEMU (no emulated Wi-Fi), so it
waits until the wired/QEMU path below is finished.

** This machine's card (concrete)

=lspci -nnk=: =00:14.3 [8086:34f0] Intel Ice Lake-LP PCH CNVi WiFi=, driver
=iwlwifi= + op-mode =iwlmvm=, host interface =wlp0s20f3=. It is an *AX201 (Wi-Fi
6)*; because it is *CNVi* (the MAC is integrated into the PCH, the RF is a
companion module) it appears as a chipset device, not an M.2 card. Its firmware
on the host is the =iwlwifi-Qu*-hr-b0-<api>.ucode= family plus a =.pnvm= blob and
the signed =regulatory.db= / =regulatory.db.p7s=.

** What it needs (four layers)

- *Kernel 802.11 stack*, all built-in (=y=; we ship no modules): =CONFIG_CFG80211=
  (config plane, driven over *nl80211 netlink*), =CONFIG_MAC80211= (soft-MAC — the
  CPU runs the 802.11 MAC; Intel cards need it), =CONFIG_WLAN= +
  =CONFIG_WLAN_VENDOR_INTEL=, =CONFIG_IWLWIFI= + =CONFIG_IWLMVM=; pulls in crypto
  (CCMP/=libarc4=) and =CONFIG_RFKILL=.
- *Firmware blob* — the radio will not come up without the =.ucode= (+ =.pnvm= +
  =regulatory.db=) loaded at probe. ~1–2 MB, proprietary, *binary only*.
- *Userland supplicant* — any WPA2/WPA3 join needs the EAPOL 4-way handshake +
  key management, i.e. =wpa_supplicant=. We ship none; hand-rolling it in Lisp is
  unrealistic (association + handshake + AES-CCMP). nl80211 is *netlink*, much
  heavier to drive than the wired static-IP ioctls of §2 — realistically shell out
  to =iw= + =wpa_supplicant=. (An *open* network could skip the supplicant.)
- *Then the §2–3 top of stack* — an IP (DHCP, since APs lease) and DNS, just over
  =wlan0= instead of =eth0=.

** The two design constraints we settled (so it stays honest)

1. *Licensing — keep it out-of-tree, exactly like the kernel.* There is *no
   source to vendor*: =iwlwifi= the driver is in-tree GPL (comes with the kernel
   we already fetch); the proprietary part is the *firmware*, which is *data, not
   code*. Intel's licence *permits binary redistribution* (that is why distros
   ship =firmware-iwlwifi=), so we *could* legally commit it — we choose not to,
   to keep this repo MIT and source-only. Instead the firmware is *fetched* (from
   =linux-firmware=, or copied from the host's =/lib/firmware=) and copied into the
   initramfs at pack time, *never committed* — one more row in the README
   licensing table, alongside =Linux kernel | GPLv2 | fetched by deps.sh; never
   redistributed here=. (Or embed it in the bzImage via =CONFIG_EXTRA_FIRMWARE= +
   =CONFIG_EXTRA_FIRMWARE_DIR=, which never touches git either.)
2. *Portability — make it opt-in and parameterized, not baked in.* In a
   no-=modprobe=, everything-=y=, frozen-image distro, Wi-Fi is *inherently*
   machine-specific — that is not a betrayal of "minimal", it is the same
   per-machine specialisation the project already does (PCI added for *this*
   laptop's keyboard in #19; the Terminus font for *this* screen). To keep the
   default build card-agnostic: put the driver options in a *separate, opt-in
   kernel config fragment* (e.g. =kernel/wifi-intel-ax201.config=) merged only on
   request; fetch the firmware *keyed by card*; and keep *credentials out of git
   entirely* — SSID/PSK live in a file on the USB's ESP or a build-time env var
   (gitignored, with a committed =*.example=), never machine- or repo-bound. The
   *pattern* is portable even though any one fragment/firmware/credential is not;
   the doc says "for a different card, swap the fragment + firmware names."

** Decision

Planning only for now. Implement *after* the wired path (§2–3) is finished and
proven in QEMU. When we do, follow the two constraints above so the repo stays
MIT/source-only and the Wi-Fi bits remain a documented, swappable overlay rather
than a one-laptop hack. This is the step that turns the box from a demo into a
machine that is actually useful.
